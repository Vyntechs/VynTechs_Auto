# 6.0 PSD Cranks-No-Start Seed — Design

**Date**: 2026-05-24
**Status**: Draft, awaiting Brandon's review
**Trigger**: Session `1bca23cb-166a-41f6-86a3-588bc3f39adf` — Angel at Young Motorsports, 2006 F-250 6.0 PSD, cranks-no-start. The AI fabricated valley-cover access procedure ("8 bolts, intake stays in place" — false on this platform) and inverted the IPR-unplug test interpretation. Both factually wrong; both violations of the system prompt's own "do not fabricate" rule.

---

## Problem

The diagnostic system has two paths:

1. **Cached-overview** (Phase 3 PR-C/A → live on `staging-interactive-diagnostics`) — when a session resolves to a known platform + symptom, the cached structured diagnostic is loaded from the DB and rendered directly. No AI in the loop. Fabrication is structurally impossible.
2. **Legacy AI tree-engine** (lives in `lib/ai/tree-engine.ts`) — fall-through when no cache hit. Generates step-by-step recommendations from corpus + retrieval + training. The prompt has a "do not state vehicle-specific specs unless retrieved/cited" rule, but it's unenforced honor-system on the LLM.

Currently seeded platforms: **2018 F-250 6.7L PSD fuel system only.**

When Angel entered a 2006 F-250 6.0 PSD with cranks-no-start, the system fell through to the legacy AI path. The AI generated confident vehicle-specific procedural claims that aren't grounded in real 6.0 PSD service procedure and would burn shop time + condemn wrong parts if followed.

## Solution

Seed the 6.0 PSD cranks-no-start case into the existing structured tables. Future sessions on this platform + complaint pair resolve to a **cache HIT** and render the verified diagnostic walk via the existing cached-overview UI. The AI is never called for cache-HIT sessions, so platform-specific fabrication is structurally impossible by construction.

Nothing else changes. No new abstractions, no new tables, no AI prompt rewrites, no verifier-layer. We're using the architecture that already exists, with the content that's currently missing.

## Scope

**In scope:**
- Platform: **2003-2007 F-250 6.0L Power Stroke** (one platform row; year range covers full production span of the 6.0 PSD)
- Symptom: **cranks-no-start** (one symptom slug)
- 14 components (HPOP, STC fitting, standpipes, oil rails, IPR valve, ICP sensor, FICM, GPCM, glow plugs, lift pump, fuel filters, injectors, oil cooler, EGR cooler)
- 11 canonical no-start tests (DTC pull, FICM voltage, battery under crank, ICP live read, IPR duty cycle, air/puff test, ICP-sensor unplug, glow plug system, fuel side, compression, cam/crank correlation)
- Decision tree (`branch_logic`) covering the major no-start branches
- 9 failure-mode patterns (`symptom_test_implications` mapping pattern → ranked candidate diagnoses)

**Out of scope (do NOT expand the PR):**
- Other 6.0 PSD symptoms (specific DTCs as primary, EGR cooler / oil cooler standalone, glow plug only, etc.)
- Other platforms (7.3L, 6.4L, 6.7L cases beyond what already exists for 2018)
- The topology-guided-diagnostic spec drafted today (parked per Brandon — "way, way, way in the future")
- PR 4 — wiring Prompts 1-4A as a production skill for auto-seeding. Follows this PR; scales the approach to other platforms without manual research per platform.
- Any change to the legacy AI tree-engine path. Sessions that don't hit cache still get whatever the AI generates — that's a separate, larger fix.

## Content sourcing

Research lives at `docs/superpowers/specs/2026-05-24-6.0-psd-cranks-no-start-research-input.md` (~5,000 words, 40+ cited sources from diesel-tech canonical sites — Bostech, Bullet Proof Diesel, DieselHub, DrivingLine, Riffraff Diesel, Ford Truck Enthusiasts, Powerstroke.org, VehicleServicePros). Every claim has ≥2 independent sources. Disagreements are named explicitly. "No verified source" markers exist where research couldn't triangulate.

**Fact-check pass before seeding:** a second-source verification agent re-verifies the high-stakes claims (test procedures, expected readings, IPR vs ICP sensor distinction, labor estimates, threshold values) against fresh sources independent of the original research. Anything that fails triangulation gets flagged for Brandon's review BEFORE the content is structured into rows.

**Brandon is the final gate.** As the master tech and domain expert, Brandon reviews the seed content (translated into a plain-English summary, not SQL) before any DB write happens. His call overrides the research.

## Safety controls (load-bearing)

1. **Project pinning.** All DB operations target the Vyntechs production project `ynmtszuybeenjbigxdyl` only. No other Supabase project gets touched. Project ID is hardcoded in every MCP call.
2. **Additive only.** Seed is INSERT statements. No DROP, no ALTER, no UPDATE of existing rows. Schema is unchanged.
3. **Reversibility via `is_retired`.** Every seeded row carries the `is_retired` flag from Phase 1's schema. If a row is later found wrong, it's marked `is_retired = true` (immediately invisible to the cached-overview loader) and replaced via the `replaced_by_id` chain. **No row is ever DELETEd in prod by this PR or its rollback path.**
4. **Three rehearsal layers before prod write:**
   - **Local `vyntechs_rehearsal` Postgres** (per Brandon's standing memory) — apply seed, run E2E, verify cache HIT renders correctly. Catches schema-shape and FK errors locally.
   - **Prod BEGIN/ROLLBACK via Supabase MCP** (per Brandon's standing memory) — proves the seed commits cleanly against current prod state, then rolls back. Zero data persisted. Catches conflicts with current prod content.
   - **Per-batch real apply** — only after both rehearsals pass and Brandon approves the batch.
5. **Per-step approval.** Each batch of INSERTs (platforms → symptoms → components → observable_properties → test_actions → branch_logic → symptom_test_implications) requires explicit Brandon approval before commit. Default = no commit. Brandon explicitly says "apply batch X" before any single batch goes live.
6. **No CI auto-apply.** This seed is NOT applied by any pipeline. Only manual, gated, in-conversation. The migration file exists in git but its application is human-driven.
7. **No `--no-verify`, no `git push --force`, no `reset --hard`.** Per Brandon's standing safety memories.

## Rollback plan

If anything is wrong after prod apply:
1. **Mark affected rows `is_retired = true`.** Immediately invisible to the cached-overview loader. Sessions that previously hit cache for the affected (platform, symptom) now fall through to the AI path — same behavior as before this PR.
2. **Replacement rows insertable later** under the same slug + `replaced_by_id` chain. History is preserved in the retired rows.
3. **Worst case** — retire the entire 6.0 PSD platform row. All seed-driven behavior reverts to legacy AI path. Net effect: we're back to where we were before this PR, with no data loss anywhere else.
4. **No row is ever DELETEd in prod** by this PR or its rollback path.

## Decision log (why this shape)

- Earlier in the brainstorm, two other shapes were considered and rejected:
  - **Labeling unverified AI claims** — rejected because Brandon's bar is "no inaccurate info on screen, ever," not "label inaccurate info as such." A warning header doesn't change that "8 bolts, intake stays" is still wrong on the page.
  - **Server-side verifier pass with claim extraction** — was the next candidate. Brandon's correction made it unnecessary: the system already has structured-data infrastructure that, when populated, makes the AI irrelevant for that case. The verifier was solving a problem the cached-overview path already solves — and only for the platforms where we have content. The verifier approach also doesn't address the IPR-vs-ICP-sensor logic-chain error.
- The chosen shape (seed content + existing cached-overview path) aligns with Brandon's standing memories:
  - "Structured-first, AI-thin" — DB does the deterministic work; AI is thin assist, never in the retrieval/matching hot path.
  - "Vetted DB preferred when present, fall through when absent" — progressive enhancement: vetted on covered (platform, symptom) pairs, fall-through (legacy AI) on uncovered.
  - "AI as tool, not source of truth" — when AI is bypassed entirely on cache HIT, the tech sees only sourced content.
  - "Trust = moat" — the structurally-impossible-to-fabricate guarantee is what makes the cached-overview path trustworthy.
