## Amendment 4 ‚Äî Failure-pattern storage (the structural decision)

**Plan assumed:** `symptom_test_implications` has `pattern_description` (text) and `ranked_diagnoses` (some structured shape) columns. It does not ‚Äî see Amendment 3.

**Resolution:** Decompose the 9 failure patterns from Section 5 of the research input across the existing schema as follows:

### Generic priority ranking (`symptom_test_implications`)
For the `cranks-no-start` symptom, rank all 11 tests by **default cranks-no-start priority** (which test to run first if no pattern context is given). One row per (symptom, test_action) pair with `priority` 1 through 11:
- 1: pull-all-dtcs
- 2: battery-crank (always rule out batteries before deep diagnostics)
- 3: ficm-voltage (cheap, ~30 sec, eliminates the cheapest no-start cause)
- 4: icp-live-read (the diagnostic-defining PID)
- 5: ipr-duty-cycle (paired with ICP read)
- 6: low-pressure-fuel
- 7: glow-plug-resistance
- 8: icp-sensor-unplug (NOT the IPR)
- 9: air-puff-test
- 10: compression
- 11: cam-crank-correlation
(Engineer can adjust this ordering when writing the SQL based on the research input's prescribed default ordering.)

Total rows: **11**. Plan said ‚â•15; the actual count is 11.

### Pattern entry points (`branch_logic`)
The 9 failure patterns become `branch_logic` rows hanging off the **`pull-all-dtcs`** test_action (the entry-point test). Each pattern is encoded as a (condition, verdict, next_action, reasoning) edge:

Example:
```sql
INSERT INTO branch_logic (slug, test_action_id, condition, verdict, next_action, routes_to_test_action_id, reasoning, source_provenance, is_retired)
SELECT
  'sd3-60psd-branch-pattern-hot-restart-progressively-worse',
  ta.id,
  'No DTCs present; complaint pattern is "cold OK, hot restart progressively worse, takes longer cranking after each restart"',
  'warn',
  'Suspect injector top O-rings (#1 candidate at 99% on aging 6.0 PSDs). Secondary suspects: STC fitting (mid/late-rail trucks), valve cover gasket.',
  (SELECT id FROM test_actions WHERE slug = 'sd3-60psd-test-icp-live-read' AND is_retired = false),
  'Pattern 1 (research input Section 5): hot-restart pattern most often = injector top O-rings on aging 6.0 PSDs. Fact-check correction: NOT STC fitting as #1.',
  'FIELD-VERIFIED', false
FROM test_actions ta WHERE ta.slug = 'sd3-60psd-test-pull-all-dtcs' AND ta.is_retired = false;
```

Total pattern-rooted `branch_logic` rows: 9. Plus the ~25 test-flow edges from the decision tree (Section 4 of research input). Total branch_logic rows: **~34**.

### Why this works
- The cached-overview UI shows the 11 priority-ranked tests for `cranks-no-start` (the user's entry point).
- When the user observes a specific pattern at test #1 (DTC pull), the branch_logic edges route them to the right next test based on the pattern observed.
- The "Pattern 1: hot-restart" narrative lives in `branch_logic.reasoning` ‚Äî exactly where prose explanation of WHY a branch is taken belongs.
- No new schema. No scope cut. The fact-check correction (Pattern 1 = injector top O-rings, NOT STC fitting) lives in the `next_action` text of the pattern's branch_logic row.

## Amendment 6 ‚Äî Resolver normalization (Task 9)

Current `lib/diagnostics/resolve-platform.ts` uses `FORD_67_PSD_MODELS.has(model)` with the set `{'f-250', 'f-350', 'f-450', 'f-550'}`. Inputs like `'F250 Super Duty'` (no dash, with suffix) currently fail this check.

For Task 9, the model normalization needs strengthening ‚Äî strip trailing suffixes like " super duty", " superduty", normalize spaces and dashes. Add this BEFORE the `.has()` check, or replace `.has()` with a normalize-then-match helper.

Suggested helper (place in `resolve-platform.ts` private scope):
```typescript
function normalizeFordModel(model: string): string {
  return model
    .toLowerCase()
    .trim()
    .replace(/\s+super\s*duty\s*$/i, '')   // strip "Super Duty" suffix
    .replace(/\s+/g, '')                    // collapse all whitespace
    .replace(/^(f)(\d{3})$/, '$1-$2')       // F250 ‚Üí f-250
}
```

Apply this normalizer to BOTH the 6.7L existing branch AND the new 6.0L branch ‚Äî surgical addition, no logic change for 6.7L.

**Heads-up:** This touches the existing 6.7L branch (Amendment 6 violates the plan's "do not modify the existing 6.7L branch" rule slightly because the normalizer is shared). Two options:
- (a) Apply the normalizer ONLY in the new 6.0L branch's matching helper (keep 6.7L unchanged). 6.7L users entering "F250 Super Duty" still fail ‚Äî but this is the existing prod behavior, so it's a no-regression change.
- (b) Apply to both ‚Äî improves 6.7L resolution but introduces a small behavioral change.

**Recommend (a)** for surgical minimum-change. Engineer should call out in the PR description that the 6.7L branch is intentionally unchanged.

## Amendment 8 ‚Äî Task 13 rehearsal-DB rollback path

The plan's rollback SQL for `vyntechs_rehearsal` uses `UPDATE platforms SET is_retired = true` ‚Äî but `platforms` has no `is_retired` column (Amendment 3). The intended rehearsal-DB rollback strategy is `pg_dump` before applying the seed, then `pg_restore` after. The plan's UPDATE-based rollback won't work.

**Replacement Task 13 rollback approach:**

```bash
# Before applying the seed (Task 13 Step 1):
pg_dump vyntechs_rehearsal -F c -f /tmp/vyntechs_rehearsal_pre_60psd_$(date +%s).dump

# Apply seed (Task 13 Step 2)

# After verifying / rolling back (Task 13 Step 6):
dropdb vyntechs_rehearsal
createdb vyntechs_rehearsal
pg_restore -d vyntechs_rehearsal /tmp/vyntechs_rehearsal_pre_60psd_*.dump
```

This is the clean rollback. The plan's UPDATE-based "soft rollback" via `is_retired` only works for tables that actually have that column (most of ours do, but not `platforms` and `symptoms`).
