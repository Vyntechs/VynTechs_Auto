## Goal

Land the **additive** data layer that the new deterministic diagram engine reads. `loadSystemTopology` already returns a rich `SystemTopology`, but it silently drops six columns that already exist in the DB (the 4 meter fields + `routesToTestActionId`/`reasoning`) and collapses `symptom_test_implications` to a presence `Set` that loses `priority`. Two genuinely-new columns (`test_actions.step_kind`, `pin_scenario_readings.is_out_of_range`) need a migration. This track surfaces all of it **additively** — no existing key renamed or removed — so every current consumer keeps typechecking and the engine tracks (T3/T7) get the exact fields they key off.

## Architecture

- **Migration 0024** adds the only two columns that don't already exist: `test_actions.step_kind text` and `pin_scenario_readings.is_out_of_range boolean`, both nullable, with a `--> statement-breakpoint` between the two `ALTER`s (PGlite suite breaks without it). Journal gets idx-26.
- **schema.ts** declares those two columns nullable. (The meter fields, `routesToTestActionId`, `reasoning`, and `priority` ALREADY exist in schema.ts from migration 0021 — confirmed; do **not** re-add them.)
- **load-system-topology.ts** widens three existing SELECTs (`testActionRows`, `branchRows`, `readingRows`), adds `step_kind` to the test-action SELECT, replaces the `implicatedIds` `Set` with an `implicatedPriorities` `Map<testActionId, priority>`, folds `is_out_of_range` into a new sibling `TopologyScenario.isOutOfRange` map, and surfaces `priority` on `TopologyTestAction`. The connection SELECT is **NOT touched** (`electricalRole`/`fromPinId`/`toPinId` already surfaced at loader lines ~241-243 — LOCKED).
- **Verdict precedence (documented, not computed here):** `isOutOfRange` authoritative → branch `verdict==="fail"` → neutral graphite default. NO prose-number parsing; numeric compare is deferred (tech_outcomes not loaded). This track only *surfaces* the signals; T3 reads them.

### Scalability bar (how this track holds it)

Everything is data-only and vocabulary-driven: the loader passes through `meterMode` / `stepKind` / `observationMethod` / `electricalRole` / `priority` / `isOutOfRange` verbatim with no per-system or per-case branching. A new system/symptom/make adds rows, never code. Missing values degrade honestly: `meterMode` etc. default to `null`, a non-implicated action gets `priority: null` (not 0, not a fabricated rank), a pin absent from `is_out_of_range` is **not** out-of-range (neutral, never red). `pinReadings` stays a `Record<string,string>` untouched, so partial/incomplete scenario data is the steady state.
